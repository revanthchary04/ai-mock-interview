# Firestore rules analysis notes

Collections found in code:

- `users`
- `interviews`
- `feedback`

Auth pattern found:

- Client Firebase Auth with email/password in `components/AuthForm.tsx`
- Server-side session cookie creation and verification in `lib/actions/auth.action.ts`
- Firestore access currently happens through the Firebase Admin SDK in server actions and route handlers

Queries found:

- `feedback.where("interviewId", "==", interviewId).where("userId", "==", userId).limit(1)`
- `interviews.orderBy("createdAt", "desc").where("finalized", "==", true).where("userId", "!=", userId).limit(limit)`
- `interviews.where("userId", "==", userId).orderBy("createdAt", "desc")`

Document shapes inferred:

- `users/{uid}` => `{ name, email }`
- `interviews/{id}` => `{ role, type, level, techstack, questions, userId, finalized, coverImage, createdAt }`
- `feedback/{id}` => `{ interviewId, userId, totalScore, categoryScores, strengths, areasForImprovement, finalAssessment, createdAt }`

Prototype attack review:

- Public list exploit: blocked by auth requirement
- Unauthorized user profile reads: blocked to owner only
- Ownership hijack on interview/feedback create: blocked by `userId == request.auth.uid`
- Ownership hijack on update: blocked by immutable field checks
- Invalid update bypass: reduced by shared validators on create/update
- PII leakage from `users`: blocked to owner only

Known limitations in this prototype:

- Firestore rules language makes deep validation of every nested `categoryScores` item awkward; this prototype validates shape at a higher level, not each nested field
- Current app primarily uses Firestore via Admin SDK, so these rules protect future client access more than current server-side operations
